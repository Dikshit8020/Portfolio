A high-speed 6-axis agricultural robot that autonomously picks seedlings from a tray and transplants them into the field at 40 seedlings per minute — 4× faster than manual labor. Designed from scratch: mechanical kinematics, STM32 firmware, stepper driver tuning, and FreeRTOS multi-axis coordination.

01 / The Problem

Why Automate Transplanting?

Manual vegetable transplanting is back-breaking work. A skilled worker transplants ~10 seedlings per minute — hunched over, in direct sunlight, for 8 hours. In India, this costs ₹500–800/day per laborer, and labor shortages during peak season can mean entire crops are delayed.

Existing transplanting machines are tractor-mounted, costing ₹2-5 lakh, and can only handle specific seedling tray formats. Our goal: build a low-cost, portable, 6-axis robot that works with standard seedling trays and achieves 40+ seedlings/minute.

Engineering Challenge
A 6-axis robot needs to coordinate 6 stepper motors simultaneously — each motor has different gear ratios, step counts, and acceleration profiles. If any axis arrives early or late, the gripper misses the seedling. The entire motion must complete in <1.5 seconds per pick-and-place cycle.
02 / The Idea

The Design Approach

We chose a 6-DOF articulated arm configuration rather than a Cartesian gantry. This gives maximum workspace flexibility — the arm can pick from a tray at one height and plant at a completely different angle. The entire system runs on a single STM32F407 MCU at 168MHz with FreeRTOS, where each joint axis runs its own RTOS task receiving target positions via queues.

03 / The Journey

Week by Week

Weeks 1–3
Mechanical Design & 3D Printing
Designed 6-DOF arm in SolidWorks. 3D-printed link brackets, bearing housings. GT2 belt + NEMA 17 for axes 1-4, NEMA 23 for base. First assembly — joints too loose, 2mm backlash.
⚡ Assembly complete. Backlash issue identified.
Weeks 4–6
Firmware v1: Bare-Metal Disaster
Wrote bare-metal step generation using Timer IRQs. Worked for 1 axis. At 6 axes, interrupt priority conflicts caused missed steps. Axis 3 would stall randomly during fast moves. Timing-critical code in main loop + 6 timer IRQs = unmanageable.
❌ Bare-metal can't handle 6 simultaneous axes
Weeks 7–9
Migration to FreeRTOS
Rewrote entire firmware using FreeRTOS. Each axis = 1 task. Coordinator task computes IK, sends target steps to each axis queue. Axes run independently, signal completion via shared event group. No more interrupt conflicts.
✓ 6 axes run smoothly. No missed steps.
Weeks 10–15
TMC5160 SPI Tuning & StealthChop
Replaced A4988 drivers with TMC5160 — SPI-configurable, StealthChop for silent operation, automatic current scaling. Spent 3 weeks tuning: IRUN, IHOLD, TPWMTHRS, SGTHRS. Reduced motor heat from 80°C to 45°C. Stall detection via SPI StallGuard flag.
✓ Silent, cool, stall-safe operation
Weeks 16–24
Gripper, IK, & Speed Optimization
Designed pneumatic gripper for seedlings. Implemented inverse kinematics (geometric approach for 6-DOF). Optimized trapezoidal acceleration profiles to minimize cycle time. Final result: 40 seedlings/min with 98% pick success rate.
✓ 40/min · 98% pick rate · TIFAN-ready
04 / Technical Deep Dive

The Firmware Architecture

// axis_task.c — FreeRTOS motor control taskembedded_c
/* One instance runs per axis (6 total).
   Receives target step count via xAxisQueue[id].
   Signals completion via xCompletionQueue. */

void vAxisTask(void *pvParams) {
    AxisHandle_t *axis = (AxisHandle_t*)pvParams;
    int32_t target_steps;

    for(;;) {
        /* Block until coordinator sends target */
        xQueueReceive(xAxisQueue[axis->id],
                      &target_steps, portMAX_DELAY);

        int32_t delta = target_steps - axis->current_steps;
        axis->direction = (delta > 0) ? DIR_CW : DIR_CCW;
        HAL_GPIO_WritePin(axis->dir_port, axis->dir_pin,
                          axis->direction);

        /* Trapezoidal acceleration profile */
        uint32_t total  = abs(delta);
        uint32_t ramp   = total / 4;
        uint32_t cruise  = total - 2 * ramp;
        uint32_t delay_us = axis->max_delay;

        for(uint32_t i = 0; i < total; i++) {
            HAL_GPIO_WritePin(axis->step_port,
                              axis->step_pin, GPIO_PIN_SET);
            delay_us_precise(delay_us);
            HAL_GPIO_WritePin(axis->step_port,
                              axis->step_pin, GPIO_PIN_RESET);
            delay_us_precise(delay_us);

            if(i < ramp)
                delay_us -= axis->accel_delta;
            else if(i >= ramp + cruise)
                delay_us += axis->accel_delta;
        }

        axis->current_steps = target_steps;
        xQueueSend(xCompletionQueue, &axis->id,
                   portMAX_DELAY);
    }
}
The RTOS Sync Problem We Solved
The coordinator task must wait for all 6 axes to finish before commanding the next waypoint. We used an EventGroup with 6 bits — each axis task sets its bit on completion. The coordinator blocks on xEventGroupWaitBits(ALL_AXES_MASK). This is cleaner than counting semaphores and handles the case where some axes have zero movement (they set their bit immediately).

07 / Learnings

What I'd Do Differently

  • Use closed-loop steppers. Open-loop NEMA motors lose steps under heavy load. Adding magnetic encoders (AS5600) would allow the firmware to detect and correct stalls in real-time, eliminating the need for conservative speed limits.
  • Move inverse kinematics to a lookup table. Computing IK on-the-fly at 168MHz works but limits cycle time. Pre-computing a 3D lookup table for all tray positions would cut IK computation from 2ms to 200µs per waypoint.
  • Add CAN bus between driver boards. Currently all 6 TMC5160s share one SPI bus, creating a bottleneck. With CAN, each driver could run on its own node, enabling truly parallel step generation.
  • Design a custom PCB earlier. We ran on breadboard + breakout boards for too long. The wire inductance caused noise on the SPI bus that corrupted TMC5160 register writes. A proper 4-layer PCB from week 8 would have saved 3 weeks of debugging.
